# 팀 공간 전환 계획

## 목표
- 팀 모집 화면에서 외부 연락 링크 중심 UX를 제거한다.
- 대신 `팀장 정보 + 팀 공간 + 진행상황 공유`를 기본 구조로 보여준다.
- 현재 프론트/백엔드 구조를 크게 흔들지 않으면서 바로 적용 가능한 1차 개선과, 이후 확장 가능한 2차 구조를 분리한다.

## 1차 MVP 구성

### 1. 팀 카드 기본 구조
- 팀장: 누가 리드하는 팀인지 바로 보이게 표시
- 팀 공간: 이 팀이 어떤 방식으로 협업하고 정리하는지 한 줄 설명
- 진행 현황: `팀 셋업 중`, `제출 준비 중`, `최종 제출 완료`처럼 현재 단계를 배지로 표시
- 최근 공유: 최근 업데이트 1~2개를 카드 안에서 바로 확인
- 다음 체크포인트: 지금 팀이 당장 맞춰야 할 우선순위를 2~3개로 정리

### 2. 팀 생성/수정 입력 방식
- 기존 `연락 링크` 입력은 제거
- 대신 `팀 소개 및 진행상황` 텍스트에 다음 내용을 함께 적도록 유도
- 현재 만들고 있는 것
- 필요한 역할
- 협업 방식
- 이번 주 진행 단계

### 3. 상세 화면 확장
- 해커톤 상세의 팀 목록에서도 같은 정보 구조를 유지
- 팀장, 팀 공간, 진행 현황, 최근 공유, 체크포인트를 같은 톤으로 반복 노출

## 현재 구조 기준 데이터 모델 제안

### 1. 이번 구현 범위에서 바로 가능한 방식
- `teams`
- 기존 `description`, `desired_roles`, `tech_tags`, `updated_at`
- 기존 `team_members`
- 기존 `submissions`, `submission_versions`

- 위 데이터를 조합해서 아래 정보를 계산형으로 만든다
- 팀장 정보
- 현재 진행 단계
- 최근 공유 메시지
- 다음 체크포인트

### 2. 다음 단계에서 확장할 데이터 모델
- `team_spaces`
- `team_id`, `owner_id`, `summary`, `working_style`, `status`, `updated_at`
- `team_updates`
- `team_id`, `author_id`, `title`, `content`, `kind`, `created_at`

- 이 구조로 가면 이후에는 계산형 UI가 아니라 실제 팀룸/업데이트 흐름으로 확장 가능하다
- 1차에서는 테이블 추가 없이 UI 전환부터 적용하고, 2차에서 실제 저장형 팀 업데이트를 붙이는 것이 리스크가 낮다

## 이번 구현에서 현실적으로 넣을 화면 요소
- 해커톤 목록 정렬은 실제 선택지가 보이는 드롭다운으로 변경
- 팀 모집 카드의 외부 연락 버튼 제거
- 팀 카드에 팀장 정보 노출
- 팀 카드에 팀 공간 설명 블록 추가
- 팀 카드에 최근 공유 1~2개 추가
- 팀 카드에 다음 체크포인트 2개 추가
- 팀 생성/수정 폼에서 연락 링크 제거
- 해커톤 상세의 팀 섹션도 같은 패턴으로 정리

## 이후 확장 우선순위
1. 팀 업데이트를 실제로 저장하는 `team_updates` 테이블 추가
2. 팀장만 업데이트 작성 가능하도록 RLS 설정
3. 팀원용 내부 댓글/짧은 대화 기능 추가
4. 제출 상태와 팀 업데이트를 연결한 운영 히스토리 화면 추가
