# Handover HQ 실서비스 전환 기획서

## 1. 목표
- 피그마 기반 정적 화면을 실제 제출 가능한 해커톤 운영 서비스로 전환한다.
- Vercel 배포와 Supabase 백엔드 구성을 기준으로 인증, 데이터 저장, 제출 흐름을 구현한다.
- 설계 문서의 필수 명세를 우선 닫고, `Next Action`, `Team Fit`, `Submit Guard`를 서비스 가치 확장 포인트로 구현한다.

## 2. 구현 범위
### 필수
- 공개 해커톤 목록 조회와 상세 조회
- 팀 모집 목록/생성/수정
- 제출 초안 저장과 최종 제출
- 랭킹/리더보드 조회
- 이메일 매직링크 기반 로그인
- Vercel SPA 라우팅 대응
- Supabase RLS 기반 권한 제어

### 선택
- GitHub OAuth 로그인
- 제출 버전별 변경 이력 비교
- 운영자 전용 상태 변경 액션
- 팀 적합도 고도화 로직

## 3. 기술 방향
- 프론트엔드: React + TypeScript + Vite + Tailwind
- 상태 관리: Zustand Persist
- 검증: Zod + React Hook Form
- 백엔드: Supabase Database + Auth + Storage + RLS
- 배포: Vercel

## 5. 화면별 수용 기준
### 홈
- 로그인 상태와 팀/제출 상태를 기준으로 `Next Action`이 달라져야 한다.
- `해커톤 보러가기`, `팀 찾기`, `랭킹 보기` CTA가 첫 화면에서 보여야 한다.
- 진행 중 해커톤 수, 모집 중 팀 수, 가장 가까운 마감이 실제 데이터에서 계산되어야 한다.

### 해커톤 목록
- 상태 필터, 검색, 정렬이 실제 목록 결과에 반영되어야 한다.
- 빈 상태, 로딩 상태, 오류 상태가 공통 UI로 제공되어야 한다.
- 상세 이동 시 실제 `slug` 기반으로 연결되어야 한다.

### 해커톤 상세
- `개요`, `안내`, `평가`, `상금`, `일정`, `팀`, `제출`, `리더보드` 섹션을 실제 데이터 기준으로 렌더링해야 한다.
- 우측 요약 카드에 팀 정책, 제출 상태, 다음 행동, 마감 정보가 표시되어야 한다.
- 제출 탭에는 체크리스트, 자동 저장, 검증 메시지, 제출 잠금이 포함되어야 한다.

### 팀 모집
- 해커톤 필터, 모집 중만 보기, 역할 필터가 실제 데이터에 반영되어야 한다.
- 인증된 사용자는 팀 생성과 수정이 가능해야 한다.
- `Team Fit`이 사용자 관심 역할과 팀 모집 역할 기준으로 계산되어야 한다.

### 랭킹
- 기간 필터가 실제 데이터 집계 기준으로 동작해야 한다.
- 상위 3팀 강조 카드와 전체 테이블이 같은 데이터 소스를 사용해야 한다.
- 제출물 링크가 유효한 URL로 연결되어야 한다.

## 6. 보안 원칙
- 브라우저에는 Supabase `anon` 키만 노출한다.
- 제출, 팀 수정, 프로필 수정은 RLS 정책으로 본인 또는 팀장만 허용한다.
- `service_role` 키는 Vercel 서버 환경 변수로만 사용한다.
- 외부 링크는 `noopener,noreferrer`를 강제한다.
- URL 검증은 클라이언트와 서버 양쪽에서 수행한다.
