# Handover HQ 실서비스 전환 기획서

## 목표
- 피그마 기반 화면 구현물을 실제 제출 가능한 서비스로 전환한다.
- Vercel 배포와 Supabase 백엔드, 안전한 Auth/RLS 구조를 함께 적용한다.
- 설계 문서의 Next Action, Team Fit, Submit Guard를 실제 동작 기능으로 구현한다.

## 현재 진단
- 현재 앱은 페이지별 UI는 구현되어 있지만 대부분 하드코딩 데이터다.
- 로그인, 팀 소속, 제출 상태, 버전 이력, 리더보드 계산, 관리자 권한이 없다.
- 상세 페이지의 동적 라우트 파라미터를 실제 데이터 조회에 사용하지 않는다.
- 제출과 팀 생성은 폼 UI만 있고 저장, 검증, 권한 제어, 이력 관리가 없다.

## 기술 방향
- 프론트엔드: React + TypeScript + Vite + Tailwind 유지
- 백엔드: Supabase Database + Auth + Storage + RLS
- 인증: 이메일 매직링크 기본, GitHub OAuth 선택
- 배포: Vercel
- 데모 정책: Supabase 환경변수가 없을 때도 읽기와 기본 체험이 가능한 로컬 데모 모드 제공

## 핵심 데이터 모델
- `profiles`: 사용자 기본 정보, 관심 역할, 표시 이름, 권한
- `hackathons`: 해커톤 메타데이터와 공개 상태, 일정, 태그, 설명
- `hackathon_sections`: 상세 페이지의 섹션별 안내/평가/제출 규칙
- `teams`: 해커톤별 팀 정보, 모집 여부, 현재 인원, 연락 수단
- `team_members`: 팀 멤버 관계와 역할
- `submissions`: 제출 현황, 초안/최종 상태, 제출 시각
- `submission_versions`: 제출 버전 이력
- `leaderboard_entries`: 점수, 제출 시각, 제출물 링크
- `activity_logs`: 생성/수정/제출 기록

## 화면별 보강
### `/`
- 히어로 CTA를 `해커톤 보러가기`, `팀 찾기`, `랭킹 보기` 3개로 정리
- 로그인 상태와 팀/제출 상태를 반영한 Next Action 카드 제공
- 최근 팀 모집과 주목할 해커톤을 실제 데이터 기반으로 연결

### `/hackathons`
- 상태, 태그, 검색, 정렬을 실제 데이터 기반으로 동작
- 빈 상태, 로딩 상태, 에러 상태를 공통 패턴으로 제공
- 카드 클릭 시 `slug` 기반 상세 페이지로 이동

### `/hackathons/:slug`
- 상단 요약 카드에 팀 정책, 제출 상태, 다음 행동, 마감 정보를 표시
- 탭을 `개요`, `안내`, `평가`, `상금`, `일정`, `팀`, `제출`, `리더보드`로 정리
- 제출 탭에서 Submit Guard를 제공한다

### `/camp`
- 해커톤/역할/모집 상태 필터를 실제 데이터로 연결
- 팀 생성과 수정, 모집 여부 전환, 연락 링크 저장 지원
- 사용자 관심 역할과 팀 모집 역할을 비교한 Team Fit 표시

### `/rankings`
- 기간 필터에 따라 실제 점수/활동 집계를 바꾼다
- 상위 3팀 강조 카드와 전체 랭킹 표를 데이터 기반으로 유지
- 제출물 링크와 해커톤 상세 링크를 연결한다

## Submit Guard
- 초안 자동 저장
- URL 형식 검증
- 필수 항목 체크리스트
- 마감 카운트다운
- 최종 제출 확인 모달
- 제출 후 수정 제한
- 제출 버전 이력 저장

## Auth / RLS 원칙
- 공개 읽기: 해커톤, 공개 팀 모집글, 공개 리더보드
- 본인만 수정: 프로필
- 팀장/운영진만 수정: 팀, 모집글, 제출물
- Storage 버킷은 제출 파일과 공개 자산을 분리한다
- 서비스 롤 키는 브라우저에 두지 않는다

## 구현 우선순위
1. 문서와 작업 로그, Vercel/Supabase 기준 정리
2. 공통 데이터 모델과 로컬 데모 저장소 구현
3. Auth 컨텍스트와 Supabase 클라이언트 추가
4. 홈/목록/상세/팀/랭킹을 실데이터 구조로 전환
5. 제출 폼과 팀 생성/수정, 검증/상태 UI 구현
6. 배포 설정, 문서, 검증, 커밋 마무리
