# Handover HQ 실서비스 전환 기획서

## 목적
- 피그마 기반 화면 구현물을 실제 제출 가능한 서비스로 전환한다.
- Vercel 배포와 Supabase Auth/Database/Storage를 기준으로 동작 구조를 만든다.
- 설계 문서의 핵심 확장 기능인 Next Action, Team Fit, Submit Guard를 실제 워크플로로 구현한다.

## 현재 진단
- 현재 앱은 정적 화면 구현 중심이며 페이지별 데이터가 하드코딩되어 있다.
- 인증, 권한, 제출 이력, 팀 생성/수정 저장, 리더보드 계산이 없다.
- 상세 라우트는 동적이지만 실제 상세 데이터는 항상 동일하다.
- 디자인 톤은 확보됐지만 색상 토큰, 포커스 상태, 빈 상태 컴포넌트의 일관성이 부족하다.

## 목표 아키텍처
- 프론트엔드: React + TypeScript + Vite + Tailwind
- 백엔드: Supabase Database + Auth + Storage + RLS
- 배포: Vercel
- 폴백: Supabase 환경변수가 없을 때는 시드 데이터 + localStorage 기반 데모 모드로 동작

## 핵심 도메인
- `profiles`: 사용자 프로필, 관심 역할, 권한
- `hackathons`: 해커톤 메타데이터, 일정, 상태
- `teams`: 해커톤별 팀 정보, 모집 상태
- `team_members`: 팀 소속과 역할
- `submissions`: 제출 초안/최종 제출 정보
- `submission_versions`: 제출 이력
- `leaderboard_entries`: 해커톤별 리더보드

## 기능 기준
### 홈
- 핵심 CTA 3개: 해커톤 보러가기, 팀 찾기, 랭킹 보기
- 실데이터 기반 지표: 진행 중 해커톤 수, 모집 중 팀 수, 가장 가까운 마감
- 사용자 상태 기반 Next Action 표시

### 해커톤 목록
- 상태/태그 필터
- 정렬: 최신순, 마감 임박순
- 시드/원격 데이터 공통 렌더링

### 해커톤 상세
- `slug` 기반 상세 조회
- 탭: 개요, 안내, 평가, 상금, 일정, 팀, 제출, 리더보드
- Submit Guard: 자동 저장, URL 검증, 체크리스트, 마감 카운트다운, 최종 제출 확인

### 팀 모집
- 팀 생성/수정 저장
- 모집 상태, 역할 필터
- 관심 역할 기반 Team Fit 표시

### 랭킹
- 기간 필터 동작
- 제출 링크 연결
- 상위 팀 강조 + 전체 표

## 보안 원칙
- 공개 조회는 해커톤, 공개 팀 모집글, 공개 리더보드로 제한
- 제출과 팀 수정은 인증 사용자 + 소속 권한 체크
- 클라이언트에는 `anon key`만 사용하고 `service role key`는 절대 브라우저에 두지 않는다
- RLS와 Storage 정책을 함께 정의한다
