첨부한 레퍼런스 이미지들의 공통 미감을 참고해서, 해커톤 운영 웹 서비스를 고충실도 웹 UI로 설계해줘.

중요:
- 레퍼런스의 분위기와 아트 디렉션만 참고하고, 구도나 결과물을 그대로 복제하지 말 것
- 이 프로젝트는 모바일 앱이 아니라 웹 플랫폼이다
- 스마트폰 앱 목업 중심으로 만들지 말고, 브라우저 프레임, 데이터 패널, 테이블, 필터, 에디토리얼 타이포 중심으로 구성할 것

프로젝트명:
Handover HQ

서비스 정의:
명세서만 받아도 참가, 팀 구성, 제출, 결과 확인까지 이어지는 해커톤 운영 플랫폼

원하는 스타일:
- 일반적인 SaaS 템플릿이 아니라, 에디토리얼 테크 감성이 강한 웹 서비스
- 전시 포스터 / 피치덱 / 프로덕트 쇼케이스와 운영 툴이 결합된 느낌
- 메인과 대표 섹션은 아트 디렉션이 강해야 하고, 내부 기능 화면은 실제 서비스처럼 사용성이 좋아야 함
- “브랜드 감도가 높은 운영 플랫폼”처럼 보여야 함

핵심 미감:
- 큰 타이포그래피
- 비대칭 레이아웃
- 조형적인 브라우저 패널 콜라주
- 벤토형 모듈 레이아웃을 일부 활용
- 화이트/아이스 그레이/차콜/네이비 기반
- 코발트 블루를 메인 포인트
- 바이올렛 또는 시그널 레드는 히어로/쇼케이스에서만 제한적으로 사용
- 얇은 보더, 강한 정보 위계, 절제된 그림자
- 실제 디자이너가 아트디렉션한 듯한 화면 구성

절대 하지 말 것:
- 전형적인 AI SaaS 대시보드 느낌
- 모든 섹션을 동일한 카드 그리드로 반복
- 과한 글래스모피즘
- 의미 없는 3D 일러스트 남발
- 모바일 앱처럼 보이는 구도
- 귀엽거나 캐주얼한 느낌
- 너무 동그랗고 말랑한 컴포넌트
- 추상적인 거창한 카피
- 영어 lorem ipsum 위주 텍스트

언어:
- 전체 한국어 UI
- 버튼, 표 헤더, 필터, 상태 문구, 헬퍼 텍스트도 모두 자연스러운 한국어
- 실제 서비스에서 볼 법한 짧고 현실적인 문장 사용

디자인 시스템 방향:
- 데스크톱 1440px 기준
- 모바일 390px 대응도 함께 고려
- 12컬럼 그리드
- 라이트 모드 중심, 일부 다크 섹션 허용
- 카드 라운드 16~20
- 히어로 미디어 패널 라운드 24~32
- 얇은 보더 중심
- 그림자 최소화
- 타이포와 레이아웃으로 완성도 표현

타이포 방향:
- 한국어 지원이 좋은 네오그로테스크 산세리프 느낌
- 제목은 크게, 짧게, 강하게
- 대형 숫자와 섹션 인덱스를 적극 활용
- 에디토리얼 레이아웃 감각 유지
- 본문은 과장 없이 단정하게

반드시 설계할 프레임:
1. Art Direction Board
2. Design System / Components
3. 메인 페이지 (/)
4. 해커톤 목록 (/hackathons)
5. 해커톤 상세 (/hackathons/:slug)
6. 팀 모집 (/camp)
7. 랭킹 (/rankings)
8. 로딩 / 빈 상태 / 오류 상태
9. 모바일 핵심 화면 요약

페이지별 요구사항:

1) Art Direction Board
- 이 프로젝트의 무드보드처럼 보이는 프레임
- 키워드, 컬러 블록, 타이포 샘플, 패널 콜라주, 브라우저 프레임 스타일, 재질감 방향 정리
- 피치덱/포트폴리오 첫 장처럼 세련되게 구성
- 너무 장식적이지 말고 실제 디자인 기준 문서처럼 보여야 함

2) Design System / Components
- 색상 토큰
- 타이포 스케일
- 버튼
- 배지
- 탭
- 필터 칩
- 카드
- 표 헤더/행
- 입력 폼
- 제출 상태 배지
- 토스트
- 확인 모달
- 빈 상태 / 오류 상태 / 스켈레톤
- 전체적으로 실무적인 UI여야 함

3) 메인 페이지 (/)
- 일반적인 랜딩페이지보다 “운영 허브의 커버 페이지”처럼 설계
- 큰 타이포로 서비스 정체성을 강하게 보여주기
- 브라우저 패널 / 리더보드 / 팀 카드 / 제출 카드가 섞인 비주얼 콜라주 구성
- 해커톤 보러가기 / 팀 찾기 / 랭킹 보기 CTA
- 진행 중 해커톤 수, 모집 중 팀 수, 가까운 마감 등 요약 지표
- 빠른 시작 영역: 지금 뭘 해야 하는지 보여주는 Next Action 카드
- 일부 섹션은 다크 배경을 써도 되지만 전체는 과하게 어둡지 않게

4) 해커톤 목록 (/hackathons)
- 상단에는 에디토리얼 헤더와 큰 제목
- 아래는 실사용 중심의 필터/정렬 바
- 상태 필터: 예정 / 진행 중 / 종료
- 태그 필터
- 정렬: 최신순 / 마감 임박순
- 카드 리스트는 전부 같은 크기 말고 리듬감 있게
- 카드에는 제목, 상태, 태그, 마감일, 종료일, 참가 규모 표시
- 정보가 잘 읽히는 운영 화면이어야 함

5) 해커톤 상세 (/hackathons/:slug)
- 이 서비스의 핵심 허브 페이지
- 상단은 포스터처럼 강렬한 커버 구간
- 제목, 상태, D-day, 태그, 핵심 CTA를 시각적으로 강하게 배치
- 요약 패널에 팀 정책, 제출 상태, 다음 행동, 마감 정보 표시
- 본문은 실무형 정보 구조로 정리
- 섹션 또는 탭:
  개요 / 안내 / 평가 / 상금 / 일정 / 팀 / 제출 / 리더보드
- 일정은 타임라인 형태
- 평가는 표/배지 기반
- 제출은 체크리스트 기반의 실무형 UI:
  기획서
  웹 페이지 URL
  GitHub 저장소 링크
  솔루션 PDF
- 제출 섹션에 자동 저장, URL 검증, 제출 상태, 최종 제출 확인 모달 포함
- 리더보드는 상위 팀 강조 + 표 구조

6) 팀 모집 (/camp)
- 탐색성과 비교가 좋아야 함
- 해커톤별 필터
- 모집 중만 보기
- 역할 필터
- 팀 카드 또는 리스트
- 팀명, 연결 해커톤, 현재 인원, 모집 상태, 찾는 역할, 소개, 연락 링크 표시
- “팀 만들기” 버튼과 생성 폼
- Team Fit 컴포넌트 포함:
  사용자의 관심 역할과 팀 모집 역할이 얼마나 맞는지 시각화
- 카드 예쁨보다 읽기 쉬움과 비교 가능성이 중요

7) 랭킹 (/rankings)
- 게임 UI처럼 만들지 말고 데이터 보드처럼 설계
- 상단 헤더는 강한 타이포
- 상위 3팀은 강조 카드
- 아래는 조밀한 랭킹 표
- 기간 필터: 최근 7일 / 30일 / 전체
- rank / 팀명 / points / 제출 시각 / 링크 등을 잘 정리
- 점수 기준 설명 카드 포함

8) 상태 화면
- 로딩 상태
- 데이터 없음
- 오류 상태
- 전체 서비스와 같은 톤으로 정리
- 너무 귀엽지 말고 실제 제품 UI처럼

레이아웃 규칙:
- 모든 섹션을 같은 카드 반복으로 처리하지 말 것
- 콘텐츠 성격에 따라 밀도와 여백을 다르게 줄 것
- 히어로/쇼케이스 영역과 운영형 UI 영역의 성격 차이를 분명히 할 것
- 타이포, 숫자, 라벨, 패널 크기 차이로 리듬을 만들 것
- 강한 비주얼은 상단/대표 영역에 집중하고, 실제 기능 영역은 정돈되게

카피 규칙:
- “서비스 경험 혁신”, “당신의 여정을 시작하세요” 같은 AI스러운 문구 금지
- 실제 제품 문장 사용
- 예:
  해커톤 보러가기
  모집 중인 팀
  제출 마감까지 2일
  초안이 저장되었습니다
  최종 제출 후에는 수정이 제한됩니다

최종 결과물 목표:
- 단순히 예쁜 시안이 아니라, 실제 출시 가능한 고급 웹 플랫폼처럼 보이게
- 심사위원이 봤을 때 설계 의도와 브랜드 감도가 동시에 느껴지게
- “전형적인 AI 생성 SaaS 화면”이 아니라 “사람 손 탄 프로덕트 디자인”처럼 보이게
